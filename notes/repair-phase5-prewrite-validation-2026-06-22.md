## Target-Sized Sparse Candidate

Writing the original 1.64 GiB image to the 107 GiB userdata device leaves its
backup GPT before the physical end. Linux still discovers both partitions, but
`sgdisk -v` reports the displaced backup header.

A local userdata-sized sparse candidate was therefore generated by:

1. copying the validated raw image into a sparse file;
2. extending the file to exactly `114898743296` bytes;
3. attaching it through a 4096-byte-sector loop device;
4. running `sgdisk -e` on that loop device only;
5. validating GPT and both filesystems;
6. converting the result to Android sparse format.

Approved candidate for further review:

```text
file=artifacts/images/xiaomi-lmi-userdata-4096-20260622-sparse.img
sparse_size=934207076
expanded_size=114898743296
block_size=4096
sha256=E1D3A3D2B5BE2A85FB9909D01045A9D0B3929748BEEC35564EDD8CBF43EA5414
```

Round-trip validation with `simg2img` confirmed:

- `sgdisk -v`: no problems found;
- partitions remain 480 MiB `pmOS_boot` and 1.2 GiB `pmOS_root`;
- filesystem UUIDs remain unchanged;
- both filesystems pass read-only e2fsck.

Offline 700 MiB sparse splitting produced chunks of `680983144` and
`253223984` bytes. Their combined round trip retained the valid GPT and both
UUIDs. This is below the bootloader's 768 MiB download limit per chunk.

The similarly named file ending in
`.REJECTED-512-SECTOR-DO-NOT-FLASH` was deliberately quarantined after a test
used the wrong sector interpretation. It must never be flashed.

## Recovery Gate

Available locally:

- current boot backup, SHA-256
  `0C06AD2ACA2AB0D510E9D9C97BA31D35A514B9A3D15850B1C4A2121E55FA5CBF`;
- skkk recovery image in `artifacts/images`;
- existing Android `boot` and `super` were not modified by this repair.

Not found:

- a LineageOS ROM ZIP suitable for sideload;
- a complete MIUI fastboot ROM;
- a complete restorable userdata image.

The A15 skkk recovery returned to fastboot and is rejected. The A12 skkk TWRP
3.6.2 image subsequently RAM-booted successfully, exposed ADB, resolved
userdata to `/dev/block/sda34`, and advertised `format data`. This closes the
minimum recovery gate for a userdata-only write; see
`notes/repair-phase5-recovery-test-2026-06-22.md`. A complete ROM remains
preferable and was not found locally.

## Proposed Write Command

This command is documented but not authorized and was not executed:

```powershell
fastboot -S 700M flash userdata artifacts/images/xiaomi-lmi-userdata-4096-20260622-sparse.img
```

It is destructive to userdata. It requires a separate explicit approval after
the recovery gate is closed.
